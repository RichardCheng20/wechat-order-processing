## 1.1 已确认决策

### 项目结构

- 前端采用**标准 uni-app + Vue 3 + TypeScript**（不使用 uni-app x）。
- 现有 `vegetable-wholesale-app/` 为 uni-app x 空壳，**废弃**，按文档重建为 `frontend/`。
- 后端目录为 `backend/`，与文档一致。

### 商户与部署

- 首版**只服务一家批发档口**（单商户），运行时 `merchant_id` 固定为 1，不开发多商户 SaaS UI 和租户管理。
- **保留多商户扩展能力**：数据表、接口、业务层均保留 `merchant_id` 字段和租户隔离设计，便于后续产品商业化时支持多个批发档口，无需重构底层。
- 已有微信小程序 AppID。
- 暂时没有云服务器和域名，本地调试没问题后再购买。

### 小程序主体

- **暂定为个人主体**，用于开发测试和早期内测。
- 后续计划升级为**个体工商户认证**，以支持正式经营、服务类目扩展和微信支付接入。
- 个人主体阶段需注意：部分类目和支付能力受限，正式上线前需完成主体变更或迁移方案评估。

### 客户与订单

- 首版客户端**不展示商品价格**；客户只需浏览可售蔬菜并按数量下单。
- 订单创建时无单价、无总额；**金额在老板送达后录价时由服务端生成**。
- 客户完成邀请绑定后，可登录小程序自助下单。
- 老客户/已绑定客户可配置**自动确认**，下单后直接进入「待分拣」，跳过「待确认」。
- 未开启自动确认的客户，下单后进入「待确认」，需老板确认。
- 一个客户档案首版只允许绑定一个微信；如需绑定多个微信，必须由商户审核。
- 邀请码默认有效期 **7 天**，过期后老板可重新生成。

### 工人与权限

- 工人不需要看到客户电话和完整地址。
- 工人端显示**客户简称 + 区域/门牌简写**（如「城南农贸 3 号门」），用于分拣和配送识别。
- 工人同时负责分拣和配送。

### 管理员

- 老板不止一个账号，支持**主管理员**和**合伙人管理员**。
- 第一个主管理员：首次微信登录时，通过**配置/白名单**自动升级。
- 合伙人管理员首版采用**固定权限子集**（可看订单、派单、录价、报表、客户和商品；不能删单、不能管理主管理员和合伙人账号），不做逐项可配置。

### 功能边界

- 首版暂时不做库存。
- 首版暂时不接微信支付。
- 第一版先做微信小程序，但 uni-app 代码结构不能阻碍后续 H5、Android、iOS 迭代。

### 技术选型（已确定）

- 鉴权：**Sa-Token**
- 数据库迁移：**Flyway**
- UI 组件：**uView Plus**

## 3. 业务规则必须先明确

### 3.1 客户绑定

已确认首版规则：

- 老板创建客户档案后生成邀请码/二维码，客户扫码或输入邀请码完成微信绑定。
- 绑定成功后，客户可登录小程序浏览商品并提交订单。
- 一个客户档案首版只绑定一个微信用户。
- 多微信绑定进入待审核状态，由老板或主管理员审核。
- 邀请码默认有效期 **7 天**，过期后老板可重新生成。
- 客户档案支持配置 **自动确认订单**（`auto_confirm_order`）：开启后，该客户下单直接进入「待分拣」；未开启则进入「待确认」。
- 客户更换微信后，由老板解绑旧微信并重新生成邀请码。

### 3.2 商品和价格

已确认首版规则：

- 客户端商品列表**不返回、不展示**任何价格字段；客户只选商品与数量。
- 老板端维护商品默认价、今日价、客户协议价，供录价时参考（不对客户暴露）。
- 录价发生在**送达之后**。
- **订单金额以老板录价后服务端计算为准**；录价完成前客户订单详情不显示金额。

### 3.3 派单和工人权限

已确认首版规则：

- 工人必须绑定微信身份，登录小程序查看任务。
- 老板手动派发订单给工人（按客户默认派发可作为后续增强）。
- 工人同时负责分拣和配送。
- 工人可修改实际分拣数量、标记缺货、标记拣完、标记送达。
- 工人不显示任何价格、金额、客户电话和完整地址字段。
- 工人端显示**客户简称 + 区域/门牌简写**（如「城南农贸 3 号门」），不显示完整地址和联系电话。

### 3.4 库存

- 首版暂时不做库存。
- 库存相关能力推迟到后续版本。

建议首版：

- 第一版不做库存扣减、库存盘点和库存预警。
- 订单完成后可以先沉淀商品出货统计，为后续库存模块做准备。

### 3.5 对账和收款

- 客户是现结、日结、周结、月结还是挂账。
- 是否需要客户在小程序看到账单。
- 首版不接微信支付。

建议首版：

- 先做收款记录和客户应收。
- 暂时不接微信支付。
- 客户账单可作为 Phase 2 加强。

## 4. AI 能力边界

首版不要把 AI 作为主链路阻塞点。

建议顺序：

1. 先完成手动下单、客户下单、派单、工人任务、录价。
2. 再接文字订单解析。
3. 再接语音转文字。
4. 再接图片 OCR。
5. 最后接大模型纠错和商品别名匹配。

原则：

- AI 只生成草稿。
- 草稿必须人工确认。
- 大模型 API Key 只放后端。
- 原始图片、语音、文字都要保存。

## 5. 第一阶段开发范围

第一阶段目标：跑通最小业务闭环，不做复杂优化。

必须完成：

- 微信登录或临时登录。
- 角色：客户、老板、工人。
- 老板创建客户档案。
- 客户邀请码/二维码绑定。
- 商品分类和商品库。
- 客户端浏览商品并提交订单。
- 老板查看订单并确认。
- 老板派单给工人。
- 工人查看被派发货物清单。
- 工人标记实际数量、缺货、拣完、送达。
- 老板录价。
- 订单金额计算。
- 今日订单统计。

暂缓：

- 微信支付。
- 完整客户账单。
- 打印。
- 多商户 SaaS。
- 库存模块。
- 利润统计。
- AI 图片/语音识别。

2. 建立前端 uni-app 骨架。
3. 设计数据库初版 SQL。
4. 跑通登录和角色身份。
