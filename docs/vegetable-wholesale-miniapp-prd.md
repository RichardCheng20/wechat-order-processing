# 蔬菜批发商户微信小程序 PRD

版本：v0.8  
日期：2026-06-12  
阶段：MVP 需求草案  
目标用户：蔬菜批发客户、蔬菜批发商户老板、工人

## 1. Problem Statement

蔬菜批发商户目前主要通过微信图片、微信文字、电话等方式接收客户订单。订单进入后，商户需要人工把客户名称、商品品类、数量、配送地点手写到纸单上，再把纸单交给工人分拣。货物分拣、配送完成后，还需要人工补录单价、金额，并在当天结束后手工汇总收入、出货量、库存变化。

这套流程的问题是：

- 接单信息分散在微信聊天、图片和电话里，容易漏记、错记、重复记录。
- 手写订单无法自动流转到分拣、配送、录价和统计环节。
- 分拣工人依赖纸单，订单状态不透明，老板不知道哪些单未拣完、哪些单已配送。
- 蔬菜价格波动频繁，录价和对账滞后，客户应收金额不清晰。
- 每日收入、客户订单量、商品出货量、库存余量没有实时统计。
- 老客户常买品类重复度高，但每次仍要从头记录，浪费时间。

## 2. Solution

开发一款面向蔬菜批发交易双方的微信小程序，核心定位是“客户下单端 + 商户订单管理端”。产品先解决每天最高频的客户下单、商户接单、分拣、配送、录价、统计闭环：

1. 客户打开小程序后看到可下单的蔬菜品类，可以通过点选商品、语音输入、手写/文字输入快速下单。
2. 老板使用商户账号进入管理端，不进入客户下单界面，而是直接看到客户订单、派单情况、分拣进度、配送进度、录价状态和经营数据。
3. 系统把客户订单商品和数量转成可执行的分拣清单，支持按订单、按商品、按客户查看。
4. 工人在小程序内查看被派发的货物清单并标记拣货进度，老板可以看到未拣完、已拣完、待配送、已完成订单。
5. 配送完成后，商户录入或批量更新单价，系统自动计算订单金额和客户应收。
6. 系统自动沉淀客户、商品、价格、库存、出货和营收数据，形成每日经营报表。

首版不追求完整 ERP，而是用最短路径替代纸单和手工统计。

## 3. 参考界面观察：鲜配送首页

根据提供的“鲜配送”首页截图，首版可以借鉴以下信息架构：

- 顶部标题为“订单”，用户进入后直接处理订单，不需要先进入复杂首页。
- 搜索框优先搜索客户名称，符合批发业务按客户找单的习惯。
- 右侧筛选入口用于按配送日期、订单状态、客户、工人等条件过滤。
- 状态 Tab 包含“全部订单 / 未拣完 / 已拣完”，适合分拣场景。
- 日期筛选突出“配送：06-08 至 06-14”，说明订单视角以配送日期为主。
- “批量”入口适合批量打印、批量标记拣货、批量录价等操作。
- 底部导航包含“订单 / 录价 / 新建 / 采购 / 我的”，中间大号“+”强调快速新建订单。

首版建议在此基础上增强“待确认、待分拣、待配送、待录价、已完成”状态，帮助商户把每天的业务进度看清楚。

## 4. Product Goals

### 4.1 业务目标

- 将客户订单从微信、电话、图片转成结构化电子订单。
- 降低商户手写抄单、重复录入、人工汇总的时间成本。
- 让分拣、配送、录价、统计围绕同一张订单流转。
- 为后续库存管理、客户对账、自动报价、采购建议打基础。

### 4.2 用户价值

- 批发客户：看到可下单蔬菜，快速提交要货清单。
- 老板：随时知道今天有多少单、多少货、多少收入、哪些单还没处理，并能接单、派单、录价、查看日报。
- 工人：只查看被派发的客户货物清单，完成分拣和配送相关操作，不查看价格。

### 4.3 衡量指标

- 订单录入耗时：首版目标较手写减少 50% 以上。
- 订单漏记率：通过订单状态和客户维度减少漏单。
- 分拣完成可视化率：每天订单都能看到是否已拣完。
- 每日经营统计生成时间：从人工汇总改为实时或一键生成。
- 商品出货统计覆盖率：订单完成后自动形成商品维度出货量。

## 5. User Personas

### 5.1 老板/店主

关注点：今天赚了多少钱、哪些客户下单、哪些货卖得多、工人有没有完成分拣和配送。

常用功能：订单总览、客户订单、派单、录价、客户差异价格、日报、应收账款。

### 5.2 批发客户

关注点：今天有哪些菜能买、怎么快速把要货清单发给商户、订单有没有被商户接收。

常用功能：浏览蔬菜品类、搜索商品、语音下单、手写/文字下单、提交订单、查看订单状态。

### 5.3 工人

关注点：老板派给我的客户要什么货、数量多少、有没有备注、是否已经拣完或送达。

常用功能：查看被派发订单、查看客户货物清单、修改实际分拣数量、标记缺货、标记拣完、标记送达。

权限限制：工人不查看商品价格、订单金额、客户欠款、收入统计和利润信息。

## 6. Scope

### 6.1 MVP 范围

MVP 必须覆盖以下闭环：

1. 客户管理
2. 商品管理
3. 客户端商品浏览和下单
4. 客户端语音输入、手写/文字输入下单
5. 商户端订单工作台
6. 订单状态流转
7. 分拣清单
8. 配送状态
9. 录价与金额计算
10. 每日经营统计
11. 工人角色权限
12. 主管理员和合伙人管理员

### 6.2 暂不进入 MVP

- 面向公开流量的生鲜商城。
- 多商户 SaaS UI 和租户自助开通（首版只服务一家批发档口，但数据层保留 `merchant_id` 扩展能力）。
- 自动读取微信聊天记录。
- 与微信支付、企业微信、第三方 ERP 深度打通。
- 完整财务系统、发票系统。
- 库存扣减、库存盘点、库存预警。
- 智能路线规划。
- 多仓库、多门店复杂调拨。
- 完全无人确认的 AI 自动下单。

## 8. Product Structure

### 8.1 端与角色

同一个微信小程序内按登录身份进入不同首页：

- 客户：进入客户下单端，核心是浏览商品、语音/手写下单、查看订单。
- 老板：进入商户管理端，核心是订单工作台、派单、录价、客户、商品和报表。
- 工人：进入工人任务端，只查看老板派发给自己的客户货物清单和配送信息。

身份判断（已确认）：

- 未登录用户可浏览公开说明或进入登录/绑定页，不直接开放完整下单能力。
- 已绑定客户身份的微信用户，登录后进入客户下单端，可浏览商品并提交订单。
- 已绑定为主管理员的微信用户，首次登录时若 openid 在配置白名单中，自动升级为主管理员。
- 已绑定为合伙人管理员或工人的微信用户，登录后进入对应商户端。
- 如果同一微信用户同时具备客户和工人身份，需要提供身份切换入口。
- 主管理员可在「我的/工人管理」里绑定工人微信身份；可添加合伙人管理员。

客户绑定（已确认）：

- 客户不做完全公开注册，由商户创建客户档案后邀请绑定。
- 商户生成客户邀请码或二维码，默认有效期 **7 天**，过期后重新生成。
- 客户扫码或输入邀请码完成微信身份绑定，绑定后可登录小程序自助下单。
- 客户绑定后，默认地址、联系人、账期、价格等级、历史订单都跟随客户档案。
- 未绑定客户只能看到基础引导页，不能提交正式订单。
- 客户档案支持配置 **自动确认订单**：开启后下单直接进入「待分拣」；未开启则进入「待确认」，需老板确认。

### 8.2 客户端底部导航

建议首版客户底部导航：

- 下单：首页商品品类、搜索、常购商品。
- 订单：查看已提交订单和状态。
- 我的：地址、联系人、历史订单。

### 8.3 商户端底部导航

建议首版商户底部导航：

- 订单：订单工作台，查看和处理订单状态。
- 派单：将客户订单分配给不同工人。
- 录价：集中处理待录价订单和今日价格。
- 新建：中间突出按钮，进入快速建单。
- 商品：商品分类、商品价格、上下架。
- 我的：工人、客户、商品、设置、数据导出。

如果后续库存和采购流程变重，可以再增加“库存”或“采购”模块。

### 8.4 客户端首页：商品下单

首页目标：客户打开小程序即可知道有哪些蔬菜可以下单，并能快速提交要货清单。

核心元素：

- 顶部搜索：搜索蔬菜名称和别名。
- 品类入口：叶菜、根茎、瓜果、菌菇、葱姜蒜、豆制品、其他。
- 常购商品：基于客户历史订单展示。
- 今日可供：商户上架的商品列表。
- 商品卡片字段：
  - 商品名称
  - 单位
  - 加入清单（数量加减）
- **不展示价格**：商户尚未录入当日成交价时，客户只需知道有哪些菜、按什么单位买、买多少；价格与金额在老板录价后才产生。
- 快捷下单：
  - 语音输入
  - 手写/文字输入
  - 复购上次订单

语音和手写输入流程：

1. 客户说出或输入要货清单。
2. 系统识别商品、数量、单位。
3. 客户确认识别结果。
4. 选择配送地址和期望送达时间。
5. 提交订单（此时订单**无单价、无总额**）：
   - 若客户档案开启自动确认 → 进入「待分拣」
   - 若未开启自动确认 → 进入「待确认」，需老板确认后进入「待分拣」
6. 订单经分拣、配送、送达后进入「待录价」；老板录价后服务端计算金额，订单进入「已完成」，客户方订单详情才可见成交金额。

### 8.5 商户端首页：订单工作台

首页目标：老板打开小程序即可知道今天哪些订单要处理。

核心元素：

- 顶部标题：订单
- 搜索框：搜索客户名称、订单号、商品名称
- 筛选按钮：配送日期、订单状态、客户、工人、是否待录价
- 状态 Tab：
  - 全部
  - 待确认
  - 待分拣
  - 分拣中
  - 已拣完
  - 配送中
  - 待录价
  - 已完成
- 日期快捷筛选：
  - 今日
  - 明日
  - 本周
  - 自定义
- 批量操作：
  - 批量派发给工人
  - 批量标记已拣完
  - 批量打印/分享分拣单
  - 批量录价
- 订单卡片字段：
  - 客户名称
  - 配送日期和时间段
  - 订单状态
  - 商品件数/品类数
  - 是否有缺货
  - 是否已录价
  - 订单金额
  - 配送地址简写

### 8.6 商户代客新建订单

入口：商户端底部中间“+”按钮。

建单方式：

- 图片识别：上传客户发来的订单截图、手写单照片、图片清单。
- 文字粘贴：粘贴客户微信文字订单。
- 语音/电话速记：接电话时边听边快速输入，也可后续接入语音转文字。
- 手动开单：选择客户后逐行添加商品和数量。
- 复制历史单：选择客户最近订单，一键复制后修改数量。

建单流程：

1. 选择或新建客户。
2. 选择订单来源。
3. 系统解析商品、数量、单位、备注。
4. 人工确认识别结果。
5. 设置配送日期、地址、联系人。
6. 保存为订单，状态进入“待分拣”。

AI/识别原则：

- 识别结果必须进入“待确认”草稿，不直接变成正式订单。
- 无法匹配的商品显示为“未识别商品”，要求人工选择或新建商品。
- 数量和单位不确定时高亮提示。
- 保留原始图片/文字，方便追溯。

### 8.7 派单

派单目标：老板把客户订单分配给不同工人处理，工人只看到自己要处理的客户货物清单。

派单方式：

- 单个订单派发：在订单详情里选择工人。
- 批量派发：在订单列表选择多张订单后派发给同一工人。
- 按客户默认派发：固定某些客户默认由某个工人处理。
- 改派：老板可把未完成订单改派给其他工人，并保留操作记录。

工人任务页（已确认）：

- 只展示被派发给当前工人的订单。
- 按客户展示货物清单。
- 显示**客户简称 + 区域/门牌简写**（如「城南农贸 3 号门」），用于分拣和配送识别。
- 可查看商品名称、下单数量、单位、备注、实际分拣数量、缺货状态。
- 不展示客户电话、完整配送地址、单价、小计、订单金额、客户欠款、日报收入和利润。

派单状态：

- 未派发：老板尚未指定工人。
- 已派发：工人可以看到任务。
- 分拣中：工人已开始处理。
- 已拣完：工人标记完成分拣。
- 已送达：如果该工人也负责配送，可标记送达。

### 8.8 分拣

分拣视角需要同时支持“按客户订单”和“按商品”。

按订单：

- 查看某客户订单的商品清单。
- 每个商品显示下单数量、单位、实际分拣数量、备注。
- 支持单项标记完成。
- 支持整单标记已拣完。

按商品：

- 汇总今日所有订单中某商品总需求。
- 显示涉及客户列表。
- 适合早上统一备货和称重。

异常处理：

- 缺货：标记缺货数量和替代方案。
- 换货：记录原商品和替代商品。
- 数量变更：记录实际数量和原因。
- 备注：例如“要嫩一点”“不要叶子黄的”。

### 8.9 配送

配送模块聚焦订单流转，不做复杂物流系统。小商户场景里，分拣和配送可能由同一个工人完成。

功能：

- 待配送列表。
- 配送状态标记。
- 标记已装车。
- 标记已送达。
- 配送备注和异常记录。

说明：首版工人不在系统内查看客户电话和完整地址。工人端仅显示客户简称和区域/门牌简写，完整地址和联系方式由老板在管理端掌握，必要时线下协调。

订单状态：

- 已拣完后进入“待配送”。
- 工人标记装车后进入“配送中”。
- 送达后进入“待录价”或“已完成”，取决于是否已录价。

### 8.10 录价与结算

蔬菜批发价格波动大，**当日成交价通常在分拣/出货/送达后才由老板确认**。客户下单时商户往往尚未录入价格，因此客户端下单阶段不展示价格；录价由老板在送达之后完成，工人同样不可见价格和金额。

功能：

- 待录价订单列表。
- 按订单录价。
- 按商品批量录入今日价格。
- 客户特殊价格：同一个商品可对不同客户设置不同价格。
- 单位价格和重量换算。
- 自动计算小计、订单总额、客户应收。
- 支持订单优惠、抹零、运费。
- 生成客户账单。

价格优先级：

1. 订单单独价格
2. 客户协议价
3. 今日商品价
4. 商品默认价

录价状态：

- 未录价：订单商品没有完整单价。
- 部分录价：部分商品缺价格。
- 已录价：所有商品有单价并生成金额。

### 8.11 统计报表

日报是 MVP 的关键价值，优先做简单但准确。

日报字段：

- 今日订单数
- 今日客户数
- 今日销售额
- 今日已收金额
- 今日应收金额
- 今日出货品类数
- 今日商品出货排行
- 待分拣订单数
- 待配送订单数
- 待录价订单数
- 缺货商品列表

维度：

- 按日期
- 按客户
- 按商品
- 按订单状态

导出/分享：

- 生成图片日报，便于微信转发给老板。
- 导出 Excel 可作为后续版本能力。

### 8.12 出货统计

首版暂时不做库存模块，只沉淀商品出货统计。

出货统计字段：

- 日期
- 商品
- 下单数量
- 实际分拣数量
- 单位
- 涉及客户数
- 涉及订单数

说明：

- 首版不做库存扣减、库存盘点、库存预警。
- 商品出货统计用于老板复盘当天卖了什么，也为后续库存和采购建议打基础。

## 9. Order State Machine

订单状态建议：

```text
草稿
  -> 待确认
  -> 待分拣
  -> 分拣中
  -> 已拣完
  -> 待配送
  -> 配送中
  -> 已送达
  -> 待录价
  -> 已完成
```

补充规则：

- 如果建单时人工已确认，可跳过“待确认”直接进入“待分拣”。
- 如果订单先录价再配送，送达后可直接进入“已完成”。
- 如果配送完成但价格未完整录入，则进入“待录价”。
- 缺货、换货、取消不作为主状态，作为异常标签展示。

## 11. Permissions

建议角色：

- 客户：浏览授权商品、按数量提交订单、查看自己的订单；**下单阶段不展示价格**；录价完成后可在订单详情查看成交金额与账单。
- 主管理员：全部权限，包括客户管理、商品管理、订单确认、派单、录价、收款、对账、合伙人账号管理和报表。
- 合伙人管理员：由主管理员授权，可查看和处理订单、派单、录价、报表，具体权限可配置。
- 工人：只查看被派发给自己的订单任务，修改实际分拣数量，标记缺货、拣完、装车和送达。

权限原则：

- 客户只能查看自己的订单、地址和账单，不能查看其他客户信息；**下单浏览商品时不展示任何价格字段**。
- 客户必须完成商户邀请或绑定后才能提交正式订单。
- 主管理员和被授权的合伙人管理员可以为同一个商品设置不同客户价格，并负责最终录价。
- 合伙人管理员首版采用**固定权限子集**（可看订单、派单、录价、报表、客户和商品；不能删单、不能管理账号），不做逐项可配置。
- 工人只能看被派发订单的客户货物清单和配送简写地址。
- 工人默认不看客户电话、完整地址、商品单价、订单金额、销售额、客户欠款、利润和采购价。
- 删除订单需要主管理员权限，并保留操作记录。

## 12. Implementation Decisions

### 12.1 技术形态

- 前端：标准 **uni-app + Vue 3 + TypeScript** 开发微信小程序，UI 组件使用 **uView Plus**。使用 HBuilderX 运行、微信开发者工具预览、Cursor 辅助编码。不使用 uni-app x。
- 后端：Java 21 + Spring Boot 3，自建后端服务，鉴权使用 **Sa-Token**，数据库迁移使用 **Flyway**。
- 数据库：MySQL 存储客户、商品、订单、订单明细、价格、收款记录等核心业务数据。
- 缓存：Redis 用于登录态、验证码、热点配置、限流和短期任务状态。
- 文件存储：使用对象存储保存商品图片、订单图片、手写图片、语音文件等；开发期可本地存储。
- 部署：云服务器 + Nginx + Docker，后续可扩展为独立 Web 管理后台。
- AI 接入：微信小程序、H5、App 均不直接调用大模型，统一通过 Java 后端封装 OCR、语音识别和大模型订单解析能力。
- 管理端：MVP 不做 Web 后台，同一个小程序按身份区分客户下单端、老板端和工人端。
- 登录鉴权：微信登录获取 openid，再绑定客户、主管理员、合伙人管理员或工人角色。第一个主管理员通过**配置/白名单**自动升级。
- 项目目录：`docs/`、`backend/`、`frontend/`（废弃现有 uni-app x 空壳）。
- 商户模型：首版运行时只服务**一家批发档口**（`merchant_id = 1`）；数据表与接口保留 `merchant_id` 租户字段，便于后续商业化支持多商户。
- 小程序主体：**暂定为个人**，后续升级个体工商户认证。

详细技术架构、模块拆分、数据表建议、接口边界和开发顺序见 `docs/technical-architecture.md`。

### 12.2 识别与人工确认

- 图片、文字、语音解析都只能生成订单草稿。
- 系统必须提供确认页，明确展示“识别原文”和“结构化结果”。
- 商品名称匹配采用商品库和别名库。
- 无法匹配商品需要人工选择、新建或忽略。

### 12.3 商品与单位

- 商品必须支持别名，例如“油麦菜/油麦”“上海青/小青菜”。
- 同一商品允许多个单位，但订单明细要记录本次单位。
- 后续可增加单位换算，例如 1 箱 = 20 斤。

### 12.4 价格与客户端可见性

批发场景下，商户常在**送达之后**才确定当日成交价；客户上午/提前下单时，老板往往尚未完成价格录入。因此定价与展示规则如下：

**客户端（下单阶段）**

- 只展示：商品名称、分类、单位、是否可售。
- **不展示**：默认价、协议价、参考价、订单金额、小计。
- 客户操作：选择商品与数量 → 提交要货清单；订单明细仅记录 `order_qty`，无单价与总额。

**订单流转（可无价格）**

- 订单允许在**无单价、无总额**的情况下完成：待确认 → 待分拣 → 分拣 → 配送 → 已送达 → **待录价**。
- 录价发生在**送达之后**，由老板完成（可参考客户协议价、今日商品价、默认价等后台维护的价格数据）。

**录价后（金额生成）**

- 老板为订单明细录入成交单价后，服务端按**实际分拣数量 × 成交单价**计算小计与订单总额。
- 订单状态进入「已完成」；客户在自己的订单详情中可见成交金额（对账单推送为 Phase 2）。

**后台价格数据（仅老板端）**

- 老板端仍维护商品默认价、今日价、客户协议价，供录价时参考与快速填充，**不向客户端商品列表接口返回**。

```text
客户浏览商品（无价格） → 选数量提交订单（无金额）
  → 老板确认（可选） → 分拣 → 配送 → 已送达
  → 老板录价 → 服务端计算金额 → 已完成（客户可见订单金额）
```

### 12.5 库存

- 首版暂时不做库存模块。
- 不做库存扣减、库存盘点和库存预警。
- 订单完成后先沉淀商品出货统计，为后续库存模块做准备。

### 12.6 报表

- 报表以订单完成和录价数据为基础。
- 出货量可以按实际分拣数量统计。
- 销售额只统计已录价订单，未录价订单单独展示为“待确认金额”。

## 13. Testing Decisions

测试应关注外部业务行为，而不是内部实现细节。

重点测试场景：

- 图片/文字建单后，订单草稿能正确进入待确认。
- 人工确认后，订单能进入待分拣。
- 工人修改实际数量后，订单明细和出货统计使用实际数量。
- 整单标记已拣完后，订单进入待配送。
- 工人标记送达后，如果未录价则进入待录价。
- 录价完成后，订单金额、客户应收、日报销售额正确更新。
- 商品缺货时，订单卡片显示异常标签，日报缺货列表能看到。
- 同一商品批量录价后，多张待录价订单同步更新价格。
- 客户特殊价格优先于今日商品价。
- 首版不测试库存扣减和库存预警。
- 未绑定客户不能提交正式订单。
- 已绑定客户只能看到自己的地址、历史订单、价格和账单。
- 老板可以创建客户档案并生成邀请绑定入口。
- 老板把订单派发给工人后，工人只能看到对应客户货物清单，不能看到客户电话、完整地址、单价和金额。

验收标准：

- 一个客户从图片下单到订单完成，全流程不需要纸质抄单。
- 老板能在订单首页看到今天所有订单的处理状态。
- 工人能独立完成分拣标记。
- 录价后系统能自动生成订单金额和当日销售统计。
- 至少支持客户、商品、订单、分拣、配送、录价、日报七个核心能力。

## 14. UX Requirements

### 14.1 设计原则

- 少打字：客户、商品、地址、常购清单尽量复用。
- 状态清楚：每张订单下一步要做什么一眼可见。
- 适合早市高峰：按钮大、列表清晰、弱网可用。
- 允许事后补全：价格、实际数量、备注可以后补，但要显示未完成状态。
- 不强迫改变习惯：先兼容微信图片、电话下单，再逐步引导客户标准化下单。

### 14.2 订单卡片建议

订单卡片示例信息：

```text
张三餐饮    今日 09:30 前送达    待分拣
8 个品类 / 42 斤    有备注
地址：城南农贸市场 3 号门
操作：查看清单 / 开始分拣
```

### 14.3 空状态

首页无订单时显示：

- 文案：暂无订单，点击下方 + 新建订单
- 操作：新建订单
- 可选：新手视频入口

### 14.4 异常标签

订单卡片或明细中需要清楚展示：

- 缺货
- 待确认
- 待录价
- 数量变更

## 19. Further Notes

首版产品最重要的不是做大而全，而是替代纸单。只要能把「微信/电话订单 → 电子订单 → 分拣 → 配送 → 录价 → 日报」跑通，就能立刻减少商户每天重复抄写和人工汇总的时间。

「鲜配送」首页的订单列表形态值得借鉴，但本项目应进一步突出批发商户的真实工作状态：待确认、待分拣、待配送、待录价。对于蔬菜批发这类高频、低毛利、时间紧的业务，首页应该像一个早市调度台，而不是普通商城后台。
